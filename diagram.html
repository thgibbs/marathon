Pi enforces on every call:

  • permission check
  • input-schema validation
  • destructive → approval
  • rate limits
  • execute w/ injected creds

↺ Approval loop (destructive actions only)

Pi's tool layer detects a destructive call → Orchestrator sets waiting_for_approval (durable wait, may last days) → prompt posted in place on the surface → human approves → Orchestrator resumes → tool layer executes. Idempotency key prevents double-execute.

↺ Feedback loop

Surface 👍 / 👎 (+ optional text) → Surface Gateway → Postgres → fed into agent memory & the eval store. Cost is tracked silently and surfaced on completion.

Key flows

1 · Slack investigation — read-only, autonomous

User @ Slack→ Slack Gateway→ Invocation Router→ Orchestrator (task + ack)→ Worker / Pi→ tool layer: read GitHub + Datadog→ model calls→ Surface Delivery→ Slack thread + 👍/👎

No approval (non-destructive) · durable checkpoints between steps · automatic retry on a transient tool failure.

2 · Document-driven execution — the headline flow

Slack request→ agent drafts design doc as a markdown PR→ people comment on PR→ agent revises→ human merges PR = approval→ agent executes plan→ progress to PR + Slack

The document is the durable plan of record · approval-by-merge · cross-surface (source = Slack, artifact = GitHub).

3 · Destructive action with approval

Pi tool layer detects destructive call→ Orchestrator: waiting_for_approval (durable, may last days)→ prompt in place on surface→ human approves→ Orchestrator resumes→ tool layer executes (injected creds)→ audit + result

Enforcement in the harness · durable human wait holds no process open · idempotency key prevents double-execute.
